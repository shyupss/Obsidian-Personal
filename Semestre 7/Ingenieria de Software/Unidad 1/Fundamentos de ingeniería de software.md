# ¿Qué es la ingeniería de software?

Es una disciplina - Aplicar teorías, métodos y herramientas donde es adecuado. Siempre trata de encontrar soluciones a problemas, incluso cuando no hay teorías ni métodos aplicables.

La ingeniería de software es una **disciplina** que busca **formalizar cómo se construye software** usando métodos, procesos y herramientas.

*“La Ingeniería del software es el estudio de los principios y metodologías para el desarrollo y mantenimiento de proyectos de desarrollo software (Zelkovitz, 1979).”*

No se trata de _hacer que funcione_, sino de **hacerlo bien, repetible y escalable**.
### Software como "capital del conocimiento"

Esto significa que el software no es sólo código. Son desiciones de diseño, conocimiento del dominio y experiencia acumulada.

# ¿Porqué existe la ingeniería de software?

En los años 60-70 se desarrollaba software sin tener en cuenta ningún tipo de parámetro de calidad:
- Software sin estándares
- Mala calidad
- Proyectos fallidos

Eso es lo que llamamos la “crisis del software”

Entonces, a causa de que los clientes buscaban cada vez más calidad nació la ingeniería de software como la conocemos hoy en día.

El **temita** es que el software es:
- intangible
- complejo
- difícil de medir

# Producto de software vs Sistema de software
#### Producto de software:
- Programa terminado
- Para muchos usuarios
- No personalizado
- Objetivo: **vender**
#### Sistema de software:
- Conjunto complejo de componentes
- Integración + arquitectura
- Hecho a medida o configurable
- Objetivo: resolver necesidades específicas

Con lo dicho, se puede decir que un sistema se compone de: Requisitos, arquitectura, integración y datos. Es decir, un producto de software es una pieza, pero un sistema de software es un ecosistema completo.

# Fundamentos para todo sistema de software

1. Debe llevarse a cabo usando un proceso de desarrollo **administrado** y **comprendido**. La organización que diseña el software necesita **planear el proceso de desarrollo.** 
2. La **confiabilidad** y el **desempeño** son importantes para todos los tipos de sistemas. No basta con que simplemente funcione.
3. Es importante **comprender y gestionar** la especificación y los **requerimientos** del software (lo que el software debe hacer). 
4. Tiene que usar de forma efectiva los recursos existentes. Se busca **eficiencia**.

# Analogía con construcción

La comparación entre un proyecto de construcción y de software nos dice que:
- **Proyecto pequeño** → una persona
- **Proyecto grande** → **múltiples** roles (arquitectos, diseñadores, etc.)
### En software:
- pequeño → script simple
- grande → sistemas complejos con:
    - arquitectos
    - diseñadores
    - QA
    - gestores

Por ende, el software complejo **requiere organización**, no solo programación.

*"El software se desarrolla o modifica con intelecto; no se manufactura en el sentido clásico. Por ende
el este no se “desgasta”, se deteriora."*

# Problemas típicos en problemas de software
### Principales:
- Incapacidad para estimar coste y esfuerzo
- Entrega tardía
- Falta de calidad del producto
### Causas profundas:
- Naturaleza no física del producto
- El software es la parte más maleable
- Problemas de comunicación con los clientes
- Problemas derivados de la intervención de distintos grupos
- Problemas de gestión (planificaciones irreales)
- Poco esfuerzo en análisis y diseño
- Poco uso de técnicas y herramientas adecuadas

# ¿Qué buscamos entonces con la ingeniería de software?

Desde entonces la Ingeniería del Software ha perseguido pasar a un estado de mayor madurez en el desarrollo del software. 

¿Cómo se llega a ese estado de mayor madurez?: A través de la definición de procesos de construcción, metodologías, etc... los cuales aseguran que el software que se desarrolla tiene la calidad requerida por el usuario.

# Ciclo de vida

Define el **orden y la secuencia** en la que se ejecutan los procesos a lo largo del tiempo. Es **orientado al producto** porque describe cómo ese producto evoluciona desde la idea hasta su retiro

No pensamos primero en “qué actividades existen”, sino en:
- qué etapas atraviesa el software
- en qué secuencia aparecen
- cómo progresa el producto entre estados

Pensemos en el software como algo que pasa por estados:
- idea / necesidad
- requisitos
- diseño
- construcción
- pruebas
- operación
- mantenimiento

Eso no significa que siempre ocurra una sola vez ni en línea recta. Puede ser secuencial, iterativo o incremental. Pero la lógica del ciclo de vida siempre responde a esta pregunta:
	***¿Cómo avanza el producto a lo largo de su existencia?***
